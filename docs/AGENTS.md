# AGENTS.md

## 역할
당신은 **HACCP 스마트 관리 시스템 v1.0**의 시니어 아키텍트이자 구현 에이전트다.

작업 전 반드시 아래 문서를 읽고 따른다.
- `docs/master_plan.md`
- `docs/spec_header.md`
- `docs/spec_approval.md`
- `docs/spec_form_engine.md`

규칙 우선순위:
1. 사용자 지시
2. `docs/` 스펙
3. `AGENTS.md`
4. 구현 편의성

---

## 최우선 가치
항상 아래 순서로 판단한다.
1. **데이터 무결성**
2. **공장 격리**
3. 결재 흐름 정확성
4. 안전하고 되돌릴 수 있는 수정
5. 인쇄 결과 일관성

---

## 절대 금지
- `factory_id` 없이 조회/저장하지 마라.
- `pb1` / `pb2` 데이터를 섞지 마라.
- `server/db.js` 밖에 SQL을 쓰지 마라.
- `doc_no`를 자동 생성하지 마라.
- 개별 양식용 신규 HTML 파일을 만들지 마라.
- 부적합(`X`)인데 개선조치 없이 제출되게 하지 마라.
- 결재 상태 전이를 임의로 바꾸지 마라.
- 백업 없이 수정하지 마라.
- 사전 계획 보고 없이 바로 수정하지 마라.
- 추측으로 수정하지 마라.

---

## 언어 규칙
- 모든 설명, 보고, 주석, 문서는 **한국어**로 작성한다.
- 기존 현업 문구, 양식명, 필드명은 요청 없으면 바꾸지 않는다.
- Windows + VS Code 환경 기준으로 안내한다.

---

## 수정 전 절차
파일 수정 전 반드시 아래 순서를 따른다.

1. 관련 문서와 관련 파일을 먼저 읽는다.
2. 구현 방향을 짧게 보고한다.
3. 수정 파일 목록을 먼저 보고한다.
4. 영향 범위를 먼저 보고한다.
5. 원본 파일을 `.backup/YYYYMMDD-HHMMSS/`에 백업한다.
6. 승인 후 수정한다.

승인 전에는 실제 코드 수정, 구조 변경, 파일 생성을 하지 마라.

---

## 기술 하네스
- 기술 스택은 `Node.js + Express + SQLite`를 유지한다.
- 공장 구분은 반드시 `factory_id` 기반으로 처리한다.
- 공장별 데이터, 권한, 조회 결과는 완전히 논리 분리한다.
- 임시 우회 로직, 공장 무시 기본값, 공용 데이터 혼합 로직을 만들지 마라.

---

## DB 규칙
- SQL은 오직 `server/db.js`에서만 작성한다.
- `routes`, `middleware`, `client`, `scripts`에 SQL을 작성하지 마라.
- 데이터 접근은 Repository Pattern으로 유지한다.
- 승인 없이 스키마를 변경하지 마라.
- 기존 컬럼 의미를 조용히 바꾸지 마라.

---

## 폼 엔진 규칙
- 모든 양식은 `log_templates`의 JSON 정의로 구동한다.
- 개별 양식용 물리 HTML 파일을 새로 만들지 마라.
- 공통 렌더링 구조를 확장해서 해결하라.
- `doc_no`는 관리자 수동 입력값만 사용한다.
- 자동 채번, 자동 증가, 임시 번호 부여를 금지한다.
- 부적합(`X`)이면 개선조치 입력 완료 전 제출을 차단한다.

---

## 결재 규칙
`docs/spec_approval.md`를 반드시 따른다.

- 2단 결재 / 3단 결재를 모두 지원한다.
- 승인자 우회, 단계 생략, 자동 승인 처리를 금지한다.
- 상태(Status) 전이 로직을 임의로 단순화하지 마라.
- 결재 로직을 수정할 때는 영향 범위를 먼저 보고한다.

---

## 인쇄 규칙
`docs/spec_header.md`를 반드시 따른다.

- 모든 양식에 표준 2단 헤더를 적용한다.
- 모든 인쇄 양식에 3칸 푸터를 적용한다.
- 푸터는 인쇄 시 하단 고정이어야 한다.
- 인쇄 문제는 임시 CSS 땜질보다 공통 인쇄 구조에서 먼저 해결한다.

---

## 주간 통합 인쇄
일 1회 기록 양식 중 5일치를 A4 1장에 담을 수 있으면, 일간 인쇄보다 **주간 통합 인쇄를 우선 적용**한다.

규칙:
- 해당 `logId`를 `client/js_print.html`의 `WEEKLY_PRINT_LOG_IDS`에 추가한다.
- 해당 주의 다음 월요일부터만 인쇄 가능하게 한다.
- 주중 중간 인쇄로 인해 기록 누락이 생기지 않게 한다.

예시:
- `si0201` 이물관리 점검표

---

## 구현 원칙
- 먼저 읽고, 그 다음 계획하고, 마지막에 수정하라.
- 가장 작은 안전한 변경을 우선하라.
- 기존 구조를 확장하라. 병렬 구조를 새로 만들지 마라.
- UI만 보고 로직을 추측하지 마라.
- 문서와 구현이 다르면 문서를 우선하고 충돌 사실을 먼저 보고하라.
- 패트롤 일지보다 **입고검사 / CCP / 결재 / 인쇄** 변경을 더 보수적으로 다뤄라.

## 보고 형식
작업 제안 시 아래 형식을 따른다.

### 구현 방향
- 무엇을 왜 바꾸는지 3~5줄

### 수정 파일
- 수정 예정 파일 경로 목록

### 영향 범위
- DB / 결재 / 인쇄 / 폼 엔진 / 공장 격리 중 영향 항목

### 리스크
- 데이터 무결성, 결재 흐름, 인쇄 깨짐, 공장 혼입 가능성

승인 전에는 실제 수정하지 마라.

---

## 완료 후 점검
수정 후 스스로 아래를 확인한다.

- `factory_id` 누락 없음
- 공장 데이터 혼합 없음
- SQL이 `server/db.js` 밖에 없음
- `doc_no` 자동 생성 없음
- 부적합 시 개선조치 강제 유지
- 결재 상태 전이 유지
- 표준 헤더/푸터 유지
- 주간 인쇄 규칙 위반 없음
- 백업 생성 완료
